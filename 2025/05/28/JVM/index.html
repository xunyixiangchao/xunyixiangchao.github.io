JVM的内存管理

Java从编译到执行

  • 解释执行

    要经过JVM翻译，速度慢一点

  • JIT执行（Java代码 翻译成 汇编码（放在codecache内存））

    速度比较快，编译时间比较长

JVM是一种规范

  • JVM的跨平台
  • JVM的语言无关性

运行时数据区

定义：Java虚拟机在执行Java程序的过程中会把它所管理的内存划分为若干个不同的数据区域

类型：线程私有与线程共享

  • 共享区：方法区，堆
  • 私有区：虚拟机栈，本地方法栈（HotSpot 合二为一了），程序计数器

Java方法的运行与虚拟机栈

程序计数器

指向当前线程正在执行的字节码指令的地址（只记录虚拟机栈的，本地方法栈值始终为null）

虚拟机栈

存储当前线程运行Java方法所需要的数据，指令，返回地址

  • 大小限制 -Xss

    创建一个线程就会创建一个虚拟机栈，基本上是1M

    -Xss1m,-Xss1024K,-Xss1048576(没有单位默认bit)

    调优：内存吃紧的情况可以调小虚拟机栈的大小

  • 栈溢出

    一个方法一个栈帧，死递归导致内存大小超过1m

栈帧

  • 局部变量表 //记录方法运行时的偏移量
  • 操作数栈
  • 动态连接
  • 完成出口 //方法结束后返回的地方

栈帧执行对内存区域的影响

使用反汇编工具输出字节码

javap -v Person.class

没有8的原因是7的操作 字节码偏移量大，占两个计数

Class回收条件 （苛刻，满足所有）

  1. class new出的所有对象都要回收掉了。
  2. 对应的类加载器也要被回收掉。
  3. 该类对应的java.lang.Class对象没有被引用
    1. 任何地方没有被引用，并且无法通过反射调用这个类的方法。
  4. 参数控制。-Xnoclassgc 关闭才行

各种引用

  • 强引用

    =

  • 软引用 SoftReference

    马上OOM的时候回收 。（图片缓存）

  • 弱引用 WeakReference

    一旦发生GC时就会被回收 （空间不够，才会GC）ThreadLocal，HashMap

  • 虚引用 PhantomReference

    监控垃圾回收器是否执行 NIO中DirectByteBuffer中的Cleaner

对象的分配策略

几乎所有的对象都在堆上分配

栈上分配 ：逃逸分析

新生代：老年代=1：2（默认）

新生代 Eden：From：To=8：1：1（默认）

对象的分配原则

  • 对象优化在Eden分配

  • 空间分配担保

    JVM担保老年代能放下，失败后发生一次fullgc

    （统计历次老年代回收释放的空间，晋级老年代时小于这个值就直接放）

  • 大对象直接进入老年代

    只有Serial和ParNew回收器才会有效， 大小超过-XX:PretenureSizeThreshold参数的设置值

  • 长期存活的对象进入老年代

    From,To区进行复制回收算法， GC计数 15次以后进入老年代

  • 动态对象年龄判定

    多个对象计算年龄和，大于survivor 期望值直接进入老年代

header中Gc为4位2进制，最大是15，达到15后进入老年代，默认15，在CMS垃圾回收器中是6

虚拟机的优化技术

栈中分配对象

逃逸分析+触发JIT（热点数据）

开启逃逸分析，并且栈里分配缓冲（TLAB），并且打开标量替换，才会在栈上分配对象（对象只在方法里用，外部没有用到。逃逸不掉）

栈不需要回收，线程执行完，栈就没了

-XX:DoEscapeAnalysis 默认开启逃逸分析 -XX:-DoEscapeAnalysis：关掉

https://docs.oracle.com/javase/8/docs/technotes/tools/unix/java.html

PS:逃逸分析有方法逃逸和线程逃逸：如果一个方法不会被其他方法调用就是方法逃逸，如果一个方法不会被其他线程访问就是线程逃逸

垃圾回收基础知识

回收新生代：minor GC

回收老年代：major GC ,OldGC

回收老年代是肯定会触发FullGC

回收算法

  • 复制算法 （主要是新生代）

    • 实现简单、运行高效
    • 没有内存碎片
    • 复用率只有一半

    Eden区的来源（Appel式的复制回收算法）

    • Appel式回收
    • 提高空间复用率和空间分配担保

  • 标记清除算法 （CMS使用）

    • 位置不连接，产生碎片（导致提前GC）
    • 可以做到不暂停

  • 标记整理算法 （标记->整理->清除）

    • 没有内存碎片
    • 指针需要移动

CMS

CMS 针对老年代的垃圾回收器（响应优先）

  1. 初始标记 ：根直接相连的对象 （STW，暂停时间短 ）

  2. 并发标记：标记除根直接相连的其他对象（并发标记）

  3. 重新标记：有变动的对象（STW，暂停所有用户线程，时间短）

  4. 并发清理：用户和GC同时进行 时间长

  5. 重置线程：

CMS中的细节

  • 预清理

    • 新生代Eden区A对象又指向了老年代的标记为垃圾的对象B，将B标记为非垃圾对象 ‘粉色’。

    • 老年代的引用发生了变化， 记录类似卡表的结构

  • 并发可中断预清理

    触发条件：Eden区已有2M（配置参数）以上

    • From或To区对象到老年代可达，导致老年代的并发标记中的引用发生变化。

    • 老年代的引用发生了变化， 记录类似卡表的结构

    中断：

    1. 循环次数（配置参数）
    2. 时间5秒（配置参数）
    3. Eden区内存使用达到50%

在并发标记和重新标记这间，节约缩短重新标记时间

CMS垃圾回收日志

CMS的问题

  • CPU敏感（核数<4基本上没法用）

  • 浮动垃圾（在并发清理时，还有新的垃圾产生）

  • 内存碎片（对象分配不下去，导致垃圾回收器退化CMS->Serial Old）重启

扩容新生代提高GC效率

扫描时间T1， 复制所用时间T2 。往往T1时间短，T2时间长

新生代容量：R
A存活时间750ms
MinorGC间隔500ms
所用时长：T1+ T2

扩容新生代：2R
A存活时间750ms
MinorGC间隔1000ms（空间增大一倍，时间拉长）
GC时，A已经不存活
所用时长：T1，没有T2了
